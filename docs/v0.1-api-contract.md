# Kember v0.1 Alpha API Contract

This document describes the public alpha contract for
`kember.openflood.org/v1alpha1`. The API is not stable. Field names, defaults,
validation rules, and lifecycle behavior may change before a stable release.

For a shorter overview, see [Alpha API](api.md).

## Goal

Kember models short-lived container work with two Kubernetes resources:

- `WorkerPool`: a platform-owned execution template and worker capacity policy.
- `TaskRun`: a tenant-owned request to execute one unit of work through a
  `WorkerPool`.

Kember does not replace the Kubernetes scheduler, Job controller, kubelet,
autoscalers, or policy systems. Its contract is limited to worker template
ownership, task assignment, terminal lifecycle state, and read-only lifecycle
projection.

## API Identity

```yaml
apiVersion: kember.openflood.org/v1alpha1
```

Both resources are namespaced:

- `WorkerPool`
- `TaskRun`

There is no conversion webhook from older experimental API groups.

## Supported Execution Matrix

| Execution mode | Lifecycle profile | Capacity model | Status |
| --- | --- | --- | --- |
| `job` | `runToCompletion` | One Kubernetes Job per TaskRun | Supported |
| `exec` | `warmLease` | Fixed warm Pod pool, one TaskRun per worker | Supported |

Any other execution mode or lifecycle profile is outside the v0.1 alpha
contract.

## WorkerPool

`WorkerPool` owns the image, command template, service account, input policy,
resource policy, security context, and task timeout policy. A `TaskRun` can
submit input and parameters, but it cannot override the WorkerPool template.

### Job WorkerPool

```yaml
apiVersion: kember.openflood.org/v1alpha1
kind: WorkerPool
metadata:
  name: echo
  namespace: kember-e2e
spec:
  execution:
    mode: job
  template:
    image: busybox@sha256:73aaf090f3d85aa34ee199857f03fa3a95c8ede2ffd4cc2cdb5b94e566b11662
    command: ["/bin/sh", "-c"]
    argsTemplate: ["echo {{input.ref}}"]
    serviceAccountName: e2e-worker
    inputPolicy:
      allowedPrefixes: ["s3://security-artifacts/project-a/"]
    resources:
      requests:
        cpu: "10m"
        memory: "16Mi"
      limits:
        cpu: "50m"
        memory: "32Mi"
  taskPolicy:
    queueTimeoutSeconds: 30
    timeoutSeconds: 60
    retentionSeconds: 60
```

### WarmLease WorkerPool

```yaml
apiVersion: kember.openflood.org/v1alpha1
kind: WorkerPool
metadata:
  name: echo-warm
  namespace: kember-warm-e2e
spec:
  execution:
    mode: exec
    commandTemplate:
      - /bin/sh
      - -c
      - 'sleep 3; test "$1" = "s3://security-artifacts/project-a/source.tar.gz"'
      - kember-task
      - "{{input.ref}}"
  lifecycle:
    profile: warmLease
    maxTasksPerWorker: 1
  capacity:
    policy: fixed
    size: 2
  template:
    image: busybox@sha256:73aaf090f3d85aa34ee199857f03fa3a95c8ede2ffd4cc2cdb5b94e566b11662
    command: ["/bin/sh", "-c"]
    args: ["touch /tmp/kember-ready; exec sleep 3600"]
    serviceAccountName: warm-worker
    inputPolicy:
      allowedPrefixes: ["s3://security-artifacts/project-a/"]
    resources:
      requests:
        cpu: "10m"
        memory: "16Mi"
      limits:
        cpu: "50m"
        memory: "32Mi"
    readinessProbe:
      exec:
        command: ["/bin/sh", "-c", "test -f /tmp/kember-ready"]
      periodSeconds: 1
  taskPolicy:
    queueTimeoutSeconds: 30
    timeoutSeconds: 60
    retentionSeconds: 60
```

### WorkerPool Spec Contract

| Field | Contract |
| --- | --- |
| `execution.mode` | `job` or `exec`. |
| `execution.commandTemplate` | Command used for `exec` WarmLease task dispatch. Supports `{{input.ref}}` and parameter substitution. |
| `lifecycle.profile` | `runToCompletion` or `warmLease`. Optional in the schema, but part of the supported mode/profile contract. |
| `lifecycle.maxTasksPerWorker` | Minimum `1`. v0.1 warm workers are single-use in the supported contract. |
| `capacity.policy` | `fixed` for WarmLease. |
| `capacity.size` | Desired warm worker pool size. |
| `template.image` | Worker image. Digest-pinned images are recommended for reproducible worker ownership. |
| `template.command` | Container command for the worker Pod or Job. |
| `template.args` | Static container args for warm worker Pods. |
| `template.argsTemplate` | Job-mode task args with input and parameter substitution. |
| `template.serviceAccountName` | ServiceAccount used by worker Pods or Jobs. TaskRun cannot override it. |
| `template.inputPolicy.allowedPrefixes` | Allowed URI prefixes for `TaskRun.spec.input.ref`. |
| `template.allowedParameters` | Optional string parameter schema with optional enum values. |
| `template.resources` | Kubernetes resource requests and limits for the worker. |
| `template.securityContext` | Optional container security context. |
| `template.readinessProbe` | Optional readiness probe. Used by WarmLease to decide assignable workers. |
| `taskPolicy.queueTimeoutSeconds` | Maximum time a TaskRun may wait before assignment. |
| `taskPolicy.timeoutSeconds` | Maximum execution timeout owned by the WorkerPool. |
| `taskPolicy.retentionSeconds` | Retention period for terminal Job resources where applicable. |

## WorkerPool Status

WarmLease pools report observed capacity:

| Field | Meaning |
| --- | --- |
| `observedGeneration` | WorkerPool generation reflected in status. |
| `capacity.desired` | Desired fixed capacity. |
| `capacity.starting` | Workers not ready yet. |
| `capacity.ready` | Ready and unleased workers. |
| `capacity.leased` | Workers assigned to a TaskRun. |
| `capacity.terminating` | Used or old workers being drained. |
| `conditions` | Kubernetes-style conditions such as `Ready`, `Progressing`, and `Degraded`. |

For Job-mode pools, capacity status is not the primary lifecycle signal.
TaskRun status is the source of task outcome.

## TaskRun

`TaskRun` references a WorkerPool in the same namespace and requests one
execution. It supplies input and optional parameters only.

```yaml
apiVersion: kember.openflood.org/v1alpha1
kind: TaskRun
metadata:
  name: echo-warm
  namespace: kember-warm-e2e
spec:
  workerPoolRef:
    name: echo-warm
  input:
    ref: s3://security-artifacts/project-a/source.tar.gz
  timeoutSeconds: 30
```

### TaskRun Spec Contract

| Field | Contract |
| --- | --- |
| `workerPoolRef.name` | Name of a WorkerPool in the same namespace. Cross-namespace references are not supported. |
| `parameters` | Optional string parameters. Keys and enum values must match `WorkerPool.spec.template.allowedParameters`. |
| `input.ref` | Input URI reference. Must match one of the WorkerPool `allowedPrefixes`. |
| `timeoutSeconds` | Per-task execution timeout. Must not exceed WorkerPool `taskPolicy.timeoutSeconds`. |
| `cancel` | When set to `true` before terminal completion, requests cancellation. Terminal TaskRuns are not re-opened. |

Kember validates `input.ref` as a reference string and passes it to the worker
contract. Kember does not download input data, upload output data, or inject
storage credentials. The trusted worker image owns its data-plane behavior.

## TaskRun Status and Phases

`TaskRun.status.phase` is monotonic and reaches at most one terminal phase.

| Phase | Meaning |
| --- | --- |
| `Pending` | TaskRun has been accepted but not dispatched. |
| `Running` | Job or warm worker execution has been dispatched. |
| `Succeeded` | Work completed successfully. |
| `Failed` | Work failed or the worker outcome could not be treated as success. |
| `TimedOut` | Queue or execution timeout was reached. |
| `Rejected` | WorkerPool, input, parameter, or timeout policy validation rejected the task. |
| `Cancelled` | Cancellation was requested before terminal completion. |

Terminal phases are final.

### Status Fields

| Field | Contract |
| --- | --- |
| `conditions` | Kubernetes-style condition entries with reason and message. |
| `resolvedTemplate` | Immutable snapshot of the selected WorkerPool execution template. |
| `jobRef` | Owned Kubernetes Job reference for Job-mode execution. |
| `workerRef` | Assigned worker Pod and Lease reference for WarmLease execution. |
| `dispatchedAt` | Time execution was dispatched. |
| `completedAt` | Time terminal status was recorded. |
| `exitCode` | Process exit code when available. |
| `result.summary` | Short textual result summary when available. |

`resolvedTemplate` records the selected worker image, command/args, execution
mode, lifecycle profile, service account, resources, security context, deadline,
retention, and template hash. It is an audit snapshot for the TaskRun and is not
an artifact storage mechanism.

## Read-only HTTP API

The Kotlin API serves read-only lifecycle projections for one configured
namespace:

```text
GET /healthz
GET /api/v1/namespaces
GET /api/v1/namespaces/{namespace}/worker-pools
GET /api/v1/namespaces/{namespace}/worker-pools/{name}
GET /api/v1/namespaces/{namespace}/task-runs
GET /api/v1/namespaces/{namespace}/task-runs/{name}
```

The alpha HTTP API does not create, update, cancel, or delete Kember resources.
It is a dashboard/query surface over Kubernetes resources.

## Security Boundary

Kember separates WorkerPool ownership from TaskRun submission:

- Platform owners define worker image, command, service account, resources, and
  security context.
- Task submitters provide input and parameters within the WorkerPool contract.
- Kubernetes RBAC remains the primary authorization mechanism for Kubernetes
  resources.

Kember is not a sandbox for untrusted images or untrusted input data. Treat
WorkerPool images as trusted platform artifacts and use Kubernetes controls such
as RBAC, service accounts, resource limits, Pod security, seccomp/AppArmor,
NetworkPolicy, non-root containers, and read-only filesystems where appropriate.

## Non-goals in v0.1 Alpha

- Stable API compatibility.
- Helm packaging.
- Multi-namespace or multi-cluster dashboard reads.
- Reusable multi-task workers.
- Autoscaling policy ownership.
- OIDC login, OPA policy adapters, Keycloak integration, KEDA adapters, or
  plugin bindings.
- Log streaming, artifact upload, artifact download, or external data-plane
  ownership.
- Exact Pod, Job, Lease, or metric timing guarantees.
